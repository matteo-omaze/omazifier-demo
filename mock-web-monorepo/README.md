# mock-web-monorepo

Stand-in for `web-monorepo`: a Next.js 15 app that composes per-market pages from omazifier. One build = one market, rooted at `/` with no market prefix in the URL.

## How omazifier is plumbed in

The entry point is a single Next.js catch-all server component:

```
app/[[...slug]]/page.tsx
```

It calls omazifier directly:

```ts
import { composePage, renderBlocks, withOfflineFallback, defaultResolveBinding } from "omazifier";
import app from "active-market";       // resolves to omazifier/markets/uk.ts at build time
import { registry } from "active-registry"; // resolves to omazifier/registry.pruned.ts

const { page, appData } = await composePage({
  app,
  path,         // from Next.js slug params
  registry,
  resolveBinding: withOfflineFallback(defaultResolveBinding),
});

renderBlocks(page.blocks, createElement, app.market)
// → wrapped in <AppShell translations={appData.translations}>
```

**`active-market` and `active-registry`** are webpack aliases set in `next.config.mjs` based on the `MARKET` env var at build time. They mean the composition and block registry for exactly one market are bundled; nothing from another market enters the output.

**`AppShell` (`components/AppShell.tsx`)** provides the two contexts every block relies on:
- `I18nProvider` — takes `appData.translations` (the market's full translation bundle from the BFF) and exposes `useTranslation()` / `t(id)` to all blocks. Blocks carry only translation key ids — no hardcoded strings.
- No omazifier knowledge — it's a plain context boundary.

**`withOfflineFallback`** is a demo-only export from the omazifier package. It wraps the default BFF resolver: if any fetch throws (network error, missing BFF), it returns bundled sample data instead. This is what allows the deployed Lambda to serve real pages with no running BFF.

## The `omazifier/` folder

```
omazifier/
├─ markets/
│   ├─ uk.ts             ← the UK composition: pages, blocks, data bindings, BFF wiring
│   ├─ de.ts             ← the DE composition (verified-entry instead of open-entry at /entry)
│   └─ fr.ts
├─ registry.ts           ← maps block ids ("hero", "draw/select", …) to web (DOM) components
└─ registry.pruned.ts    ← generated by omazifier-prune at build; only blocks used by the target MARKET
```

`markets/*.ts` are the only per-market source files. Everything else — blocks, components, contexts — is market-agnostic.

## Build scoping

```sh
MARKET=uk npm run build    # → dist/uk/  (uk composition + uk-pruned registry)
MARKET=de npm run build    # → dist/de/  (nothing from uk leaks in)
```

For Lambda deployments, `open-next.config.ts` sets `NO_DIST_DIR=1` during its build step so Next.js writes to `.next/` (where OpenNext expects it) rather than `dist/<market>/`.

## Deployed staging

UK is live at **https://omazifier-uk.stg.omaze.co.uk/** (AWS staging, eu-west-2 — Lambda + CloudFront + Route53 via CDK). The Lambda runs without a BFF; `withOfflineFallback` covers all data bindings.

To redeploy:

```sh
cd mock-web-monorepo
AWS_PROFILE=AdministratorAccess-866641491241 AWS_ACCOUNT_ID=866641491241 AWS_REGION=eu-west-2 \
  npm run infra:deploy:uk
```

## Run locally

```sh
# from omazifier-demo/ root
npm install
cd ../omazifier && npm run build    # compile the engine first

# Terminal 1 — mock BFF (optional; the app works without it via withOfflineFallback)
npm run dev:bff                     # http://localhost:4000

# Terminal 2 — pick a market
cd mock-web-monorepo
MARKET=uk npm run dev               # http://localhost:3005
MARKET=de npm run dev               # http://localhost:3006
MARKET=fr npm run dev               # http://localhost:3007
```
